# Developer guide

## Prerequisites

- Flutter SDK (see `pubspec.yaml` SDK constraint).
- For Firebase: a Firebase project, `flutterfire_cli`, and platform files (`google-services.json`, `GoogleService-Info.plist`) as generated by `flutterfire configure`.

## Run the app

```bash
flutter pub get
flutter gen-l10n   # if you change ARB files (often automatic on build)
flutter run
```

If Firebase is not configured, the app automatically falls back to **local** auth + journal storage.

## Add a new feature

1. **Domain (if needed):** Add entities under `lib/domain/entities/`, repository methods under `lib/domain/repositories/`, and use cases under `lib/domain/usecases/`.
2. **Data:** Extend models and repository implementations under `lib/data/` so all new fields are mapped for Firestore and (if applicable) local JSON.
3. **Providers:** Expose new use cases or streams in `lib/core/di/providers.dart`.
4. **UI:** Add screens under `lib/features/<feature>/presentation/` and register routes in `lib/app/app_router.dart`.
5. **Strings:** Add keys to `lib/l10n/app_en.arb` and `app_tr.arb`, then run `flutter gen-l10n`.

Keep widgets free of business rules: parse input in the pageŌĆÖs controller logic or delegate to use cases.

## Connect a new API

1. Define a **datasource** (e.g. `lib/data/datasources/remote/my_api_datasource.dart`) that performs HTTP/WebSocket calls.
2. Implement a **repository** class under `lib/data/repositories/` that satisfies the existing **domain** `*Repository` interface (or add a new interface in domain first).
3. In `main.dart`, construct the new repository and pass it through `ProviderScope(overrides: [...])` instead of (or alongside) the Firebase/local implementation.

The UI should not change if the domain contract stays stable.

## Replace Firebase

1. Implement `JournalRepository` (and optionally `AuthRepository`) with your backend.
2. In `main.dart`, remove or bypass the `Firebase.initializeApp` block and construct your repositories.
3. Keep Firestore-specific code confined to `lib/data/repositories/firebase_journal_repository.dart` (and related models) so deletion is straightforward.

Optional: introduce an enum or remote config flag (similar to `AppBackend`) to switch implementations at runtime.

## Google / Apple sign-in

`AuthRepository.signInWithGoogle` / `signInWithApple` are **intentionally unimplemented** on the Firebase repository. To enable them:

1. Add `google_sign_in` and/or `sign_in_with_apple` to `pubspec.yaml`.
2. Configure OAuth in the Firebase console and platform projects.
3. Implement the methods in `FirebaseAuthRepository` and call `FirebaseAuth` credential linking as per Firebase docs.

## Where to change UI

- **Global look:** `lib/core/theme/app_theme.dart`, `app_colors.dart`.
- **Shared chrome:** `lib/core/widgets/archive_app_bar.dart`, `lib/features/shell/main_shell.dart`.
- **Per screen:** `lib/features/*/presentation/*_page.dart`.

## Firestore rules (sketch)

Restrict reads/writes to the authenticated user:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/journalEntries/{entryId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

Adjust for your security model (custom claims, etc.).
